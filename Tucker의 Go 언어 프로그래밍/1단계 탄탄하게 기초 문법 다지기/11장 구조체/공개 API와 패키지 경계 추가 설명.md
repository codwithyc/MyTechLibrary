## 공개 API와 패키지 경계 추가 설명

구조체를 배우면 필드 이름을 대문자로 쓸지 소문자로 쓸지 결정해야 하는 순간이 생깁니다.  
이 선택은 단순한 이름 취향이 아니라, 패키지 밖에 무엇을 공개할지 정하는 문제입니다.

이번 추가 설명은 구조체 문법 자체보다 Go에서 공개 API를 어떻게 생각하는지 정리합니다.

### 대문자로 시작하면 패키지 밖에 공개된다
---

Go에서는 이름이 대문자로 시작하면 다른 패키지에서 접근할 수 있습니다.

```go
type User struct {
    Name string
    age  int
}
```

`Name`은 외부 패키지에서 접근할 수 있지만, `age`는 같은 패키지 안에서만 사용할 수 있습니다.

이 규칙은 구조체 필드뿐 아니라 함수, 타입, 변수, 상수에도 똑같이 적용됩니다.

### 공개된 이름은 약속이 된다
---

한 번 공개한 이름은 다른 코드가 사용할 수 있습니다.

```go
user.Name = "kim"
```

그래서 공개 필드를 나중에 없애거나 의미를 바꾸면, 그 필드를 쓰던 코드가 깨질 수 있습니다.  
공개된 이름은 패키지 사용자와 맺은 약속에 가깝습니다.

처음부터 모든 필드를 대문자로 열어두기보다, 외부에서 정말 직접 접근해야 하는 값인지 생각하는 습관이 필요합니다.

### 내부 상태는 메서드로 다루게 할 수 있다
---

필드를 숨기고 메서드로 접근하게 만들 수도 있습니다.

```go
type User struct {
    name string
}

func NewUser(name string) User {
    return User{name: name}
}

func (u User) Name() string {
    return u.name
}
```

이렇게 하면 내부 필드 이름이나 저장 방식이 바뀌어도 외부 코드는 `Name()` 메서드를 계속 사용할 수 있습니다.

Go에는 클래스의 private 키워드는 없지만, 패키지 경계를 기준으로 공개와 비공개를 나눕니다.

### 패키지는 캡슐화의 단위다
---

Go에서 캡슐화는 주로 패키지 단위로 이루어집니다.

같은 패키지 안에서는 소문자 이름에도 접근할 수 있고,  
다른 패키지에서는 대문자 이름만 접근할 수 있습니다.

그래서 패키지를 어떻게 나눌지는 공개 범위를 어떻게 나눌지와 연결됩니다.

```text
user/
    user.go
    password.go
```

같은 `user` 패키지 안의 파일들은 서로 내부 구현을 공유할 수 있습니다.  
반대로 외부 패키지에는 공개된 타입과 함수만 보여줍니다.

### 주석은 공개 API의 일부다
---

Go에서는 공개된 이름에 설명 주석을 붙이는 것이 좋습니다.

```go
// User represents an application user.
type User struct {
    Name string
}
```

공개 API의 주석은 문서로 노출됩니다.  
따라서 단순한 메모가 아니라, 패키지 사용자가 읽는 설명이라고 생각해야 합니다.

한국어 학습 노트에서는 한국어로 설명해도 괜찮지만, 실제 공개 패키지를 만든다면 영어 주석이 더 일반적입니다.

### 정리
---

구조체를 배울 때는 필드 선언 문법만 보지 말고, 공개 범위까지 함께 생각해야 합니다.

정리하면 다음과 같습니다.

- Go에서는 대문자로 시작하는 이름이 패키지 밖에 공개됩니다.
- 공개된 이름은 다른 코드와의 약속이 됩니다.
- 내부 상태는 소문자 필드로 숨기고 메서드로 다루게 할 수 있습니다.
- Go의 캡슐화 단위는 주로 패키지입니다.
- 공개 타입과 함수의 주석은 문서의 일부가 됩니다.

구조체는 데이터를 묶는 도구이면서, 패키지 바깥에 어떤 사용법을 허용할지 정하는 API 설계 도구이기도 합니다.
