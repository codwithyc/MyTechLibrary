## escape analysis와 메모리 생명주기 추가 설명

포인터를 배우면 자연스럽게 스택과 힙, 메모리 주소, 값의 생명주기를 궁금해하게 됩니다.  
Go에서는 개발자가 직접 스택과 힙을 선택하지 않고, 컴파일러가 값을 어디에 둘지 판단합니다.

이번 추가 설명은 포인터 문법 자체보다 Go가 메모리 생명주기를 어떻게 다루는지 큰 그림을 정리합니다.

### 지역 변수의 주소를 반환해도 된다
---

Go에서는 함수 안에서 만든 변수의 주소를 반환할 수 있습니다.

```go
func NewCounter() *int {
    count := 0
    return &count
}
```

다른 언어를 배운 적이 있다면 이 코드가 위험해 보일 수 있습니다.  
하지만 Go에서는 안전합니다.

컴파일러가 `count`가 함수 밖에서도 필요하다는 사실을 알면, 그 값을 적절한 위치에 보관합니다.

### escape analysis는 값이 함수 밖으로 나가는지 분석한다
---

escape analysis는 값이 현재 함수 범위를 벗어나서도 살아 있어야 하는지 판단하는 컴파일러 분석입니다.

```go
func Value() int {
    n := 10
    return n
}

func Pointer() *int {
    n := 10
    return &n
}
```

`Value()`의 `n`은 값만 반환됩니다.  
`Pointer()`의 `n`은 주소가 반환되므로 함수가 끝난 뒤에도 필요합니다.

이런 차이를 컴파일러가 분석해 메모리 배치를 결정합니다.

### 스택과 힙은 성능 관점에서 중요하지만 처음부터 집착하지 않는다
---

일반적으로 스택 할당은 빠르고, 힙 할당은 가비지 컬렉션의 관리 대상이 됩니다.  
그래서 성능을 깊게 다룰 때는 어떤 값이 힙으로 escape하는지 중요할 수 있습니다.

하지만 입문 단계에서는 "포인터를 쓰면 무조건 느리다"처럼 단순하게 외우면 안 됩니다.  
Go 컴파일러는 생각보다 많은 최적화를 수행합니다.

처음에는 다음 정도만 기억하면 충분합니다.

- 값이 함수 안에서만 필요하면 함수 안에서 관리될 가능성이 큽니다.
- 함수 밖에서도 필요하면 더 오래 살아 있어야 합니다.
- 정확한 배치는 컴파일러가 판단합니다.

### go build -gcflags=-m으로 힌트를 볼 수 있다
---

컴파일러의 escape analysis 결과는 다음 명령으로 일부 확인할 수 있습니다.

```bash
go build -gcflags=-m
```

출력에는 어떤 값이 heap으로 이동하는지에 대한 메시지가 나올 수 있습니다.

이 명령은 처음부터 매번 사용할 필요는 없습니다.  
하지만 포인터와 메모리를 배우면서 "컴파일러가 이런 분석을 하는구나"를 확인하기에는 좋은 도구입니다.

### 가비지 컬렉션이 메모리를 회수한다
---

Go는 가비지 컬렉션을 제공합니다.  
더 이상 사용되지 않는 힙 메모리는 런타임이 추적해서 회수합니다.

그래서 C처럼 직접 `free`를 호출하지 않습니다.  
대신 불필요하게 오래 참조를 유지하면 메모리가 회수되지 않을 수 있습니다.

```go
var cache = map[string][]byte{}
```

이런 전역 캐시에 큰 데이터를 계속 담아두면, 프로그램이 그 데이터를 계속 필요하다고 판단할 수 있습니다.  
Go가 메모리를 자동으로 관리해도, 참조를 어떻게 유지하는지는 개발자가 신경 써야 합니다.

### 정리
---

포인터를 배우는 시점에는 Go의 메모리 관리 방식도 함께 큰 그림으로 알아두면 좋습니다.

정리하면 다음과 같습니다.

- Go에서는 지역 변수의 주소를 반환해도 안전합니다.
- 컴파일러는 escape analysis로 값의 생명주기를 분석합니다.
- 스택과 힙 배치는 개발자가 직접 지정하지 않습니다.
- `go build -gcflags=-m`으로 escape 관련 힌트를 볼 수 있습니다.
- Go는 가비지 컬렉션을 제공하지만, 불필요한 참조를 오래 유지하지 않는 습관은 필요합니다.

포인터는 단순히 주소를 다루는 문법이 아니라, 값이 어디까지 살아 있어야 하는지 생각하게 만드는 문법입니다.
