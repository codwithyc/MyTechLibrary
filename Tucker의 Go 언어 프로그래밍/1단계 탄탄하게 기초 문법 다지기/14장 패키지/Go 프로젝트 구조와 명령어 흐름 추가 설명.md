## Go 프로젝트 구조와 명령어 흐름 추가 설명

패키지를 배우면 이제 Go 파일 하나를 실행하는 단계에서 벗어나, 여러 폴더와 여러 패키지로 프로젝트를 구성하게 됩니다.  
이때 중요한 것은 패키지 문법뿐 아니라 Go 도구가 프로젝트를 어떻게 바라보는지 이해하는 것입니다.

이번 추가 설명은 패키지 문법 자체보다 Go 프로젝트 구조와 자주 쓰는 명령어 흐름을 정리합니다.

### go.mod가 프로젝트의 기준점이다
---

Go Modules 환경에서는 `go.mod` 파일이 있는 곳이 모듈의 루트가 됩니다.

```text
myapp/
    go.mod
    main.go
    internal/
    pkg/
```

Go 명령어는 이 `go.mod`를 기준으로 모듈 경로와 의존성을 파악합니다.  
그래서 프로젝트를 만들 때는 먼저 적절한 위치에서 `go mod init`을 실행하는 것이 중요합니다.

```bash
go mod init example.com/myapp
```

### 패키지 경로는 폴더 구조와 연결된다
---

모듈 경로가 `example.com/myapp`이고, 하위에 `config` 폴더가 있다면 import 경로는 보통 다음처럼 됩니다.

```go
import "example.com/myapp/config"
```

즉, Go에서 폴더 구조는 단순한 정리용 디렉터리가 아니라 패키지 경로와 연결됩니다.

폴더를 나눌 때는 "보기 편하게"만 생각하기보다, 이 코드가 어떤 패키지로 사용될지 함께 생각해야 합니다.

### cmd 디렉터리는 실행 프로그램을 나눌 때 자주 쓴다
---

Go 프로젝트에서 실행 프로그램이 여러 개라면 `cmd` 디렉터리를 사용하는 구조를 자주 볼 수 있습니다.

```text
myapp/
    cmd/
        server/
            main.go
        worker/
            main.go
    internal/
        service/
```

`cmd/server`는 서버 실행 파일, `cmd/worker`는 워커 실행 파일처럼 역할을 나눌 수 있습니다.

입문 단계에서 꼭 이 구조를 써야 하는 것은 아니지만, 다른 Go 프로젝트를 읽을 때 자주 만나는 패턴입니다.

### internal 디렉터리는 내부 구현을 숨긴다
---

`internal` 디렉터리 아래의 패키지는 정해진 범위 밖에서 import할 수 없습니다.

```text
myapp/
    internal/
        auth/
```

이 구조는 "이 코드는 이 프로젝트 내부 구현이므로 외부에서 가져다 쓰지 말라"는 의도를 Go 도구 차원에서 표현합니다.

패키지 공개 범위를 관리할 때 `internal`은 매우 Go다운 도구입니다.

### go list로 Go가 보는 패키지를 확인할 수 있다
---

Go가 현재 프로젝트의 패키지를 어떻게 인식하는지 확인하고 싶다면 `go list`를 사용할 수 있습니다.

```bash
go list ./...
```

이 명령은 현재 모듈 아래의 모든 패키지를 나열합니다.  
패키지 구조가 헷갈릴 때 Go 도구가 실제로 어떤 패키지를 보고 있는지 확인하는 데 도움이 됩니다.

### 자주 쓰는 명령어 흐름
---

Go 프로젝트를 다룰 때 자주 쓰는 기본 명령어 흐름은 다음과 같습니다.

```bash
go fmt ./...
go test ./...
go build ./...
```

- `go fmt ./...`  
  하위 패키지까지 코드 포맷을 정리합니다.

- `go test ./...`  
  하위 패키지까지 테스트를 실행합니다.

- `go build ./...`  
  하위 패키지까지 빌드 가능한지 확인합니다.

이 세 가지 명령은 작은 프로젝트에서도 습관처럼 익혀두면 좋습니다.

### 정리
---

패키지를 배울 때는 파일을 나누는 문법만 보지 말고, Go 도구가 프로젝트를 어떻게 해석하는지도 함께 봐야 합니다.

정리하면 다음과 같습니다.

- `go.mod`는 Go 프로젝트의 기준점입니다.
- 폴더 구조는 패키지 import 경로와 연결됩니다.
- `cmd`는 여러 실행 프로그램을 나눌 때 자주 쓰입니다.
- `internal`은 내부 구현을 숨길 때 사용합니다.
- `go list ./...`로 Go가 인식하는 패키지를 확인할 수 있습니다.
- `go fmt ./...`, `go test ./...`, `go build ./...` 흐름은 기본 점검 루틴으로 좋습니다.

이 관점을 가지면 Go 프로젝트를 파일 묶음이 아니라, Go 도구가 이해하는 패키지와 모듈의 구조로 볼 수 있습니다.
