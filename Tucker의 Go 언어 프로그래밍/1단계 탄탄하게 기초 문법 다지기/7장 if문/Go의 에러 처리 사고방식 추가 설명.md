## Go의 에러 처리 사고방식 추가 설명

`if`문을 배우면 조건에 따라 흐름을 나누는 방법을 알게 됩니다.  
Go에서는 이 `if`문이 에러 처리와 매우 자주 연결됩니다.

이번 추가 설명은 `if` 문법 자체가 아니라, Go가 실패를 어떻게 표현하고 다루는지에 대한 사고방식을 정리합니다.

### Go는 실패를 값으로 돌려준다
---

Go에는 예외를 중심으로 코드를 작성하는 문화가 강하지 않습니다.  
대신 많은 함수가 마지막 반환값으로 `error`를 돌려줍니다.

```go
value, err := strconv.Atoi("abc")
if err != nil {
    fmt.Println("변환 실패:", err)
    return
}

fmt.Println(value)
```

이 방식은 실패 가능성을 숨기지 않고 함수 호출 결과에 직접 드러냅니다.

### err != nil은 Go 코드의 기본 리듬이다
---

Go 코드를 읽다 보면 다음 형태를 아주 자주 보게 됩니다.

```go
result, err := DoSomething()
if err != nil {
    return err
}
```

처음에는 반복이 많아 보일 수 있습니다.  
하지만 이 패턴은 "실패했으면 여기서 처리하고, 성공했으면 아래로 진행한다"는 흐름을 분명하게 만듭니다.

즉, Go의 에러 처리는 특별한 예외 흐름이 아니라 일반적인 값 검사 흐름입니다.

### 에러를 무시하지 않는 습관이 중요하다
---

Go에서는 반환된 에러를 사용하지 않으면 컴파일러가 직접 막지는 않습니다.

```go
value, _ := strconv.Atoi("abc")
fmt.Println(value)
```

`_`를 사용하면 에러를 버릴 수 있습니다.  
하지만 학습 단계에서는 에러를 버리는 습관을 들이지 않는 것이 좋습니다.

에러를 무시하면 프로그램이 왜 잘못된 값으로 동작하는지 나중에 찾기 어려워집니다.

```go
value, err := strconv.Atoi("abc")
if err != nil {
    fmt.Println(err)
    return
}
```

에러를 제대로 확인하는 것이 Go다운 안정성의 출발점입니다.

### 실패를 먼저 처리하면 정상 흐름이 읽기 쉬워진다
---

Go 코드에서는 실패 조건을 먼저 처리하고 함수를 빠져나오는 형태가 자주 사용됩니다.

```go
func PrintAge(input string) error {
    age, err := strconv.Atoi(input)
    if err != nil {
        return err
    }

    if age < 0 {
        return fmt.Errorf("나이는 음수일 수 없습니다")
    }

    fmt.Println(age)
    return nil
}
```

이렇게 작성하면 정상 흐름이 아래로 자연스럽게 이어집니다.  
중첩된 `if`를 깊게 만들지 않아도 됩니다.

### 에러 메시지는 사용자보다 개발자를 먼저 돕는다
---

학습 중에는 에러 메시지를 그냥 출력하는 경우가 많습니다.

```go
fmt.Println(err)
```

하지만 프로그램이 커질수록 에러 메시지는 문제를 추적하는 단서가 됩니다.  
어떤 입력에서, 어떤 작업 중 실패했는지를 함께 남기는 것이 좋습니다.

```go
return fmt.Errorf("나이 변환 실패: %w", err)
```

`%w`는 기존 에러를 감싸서 상위로 전달할 때 사용합니다.  
초반에는 어렵게 느껴질 수 있지만, Go의 에러는 단순 문자열이 아니라 추적 가능한 값이라는 점을 알아두면 좋습니다.

### 정리
---

Go에서 에러 처리는 별도의 고급 문법이 아니라, 값을 확인하고 흐름을 나누는 기본 습관입니다.

정리하면 다음과 같습니다.

- Go는 실패를 `error` 값으로 돌려주는 경우가 많습니다.
- `err != nil` 패턴은 Go 코드의 기본 흐름입니다.
- 학습 단계에서는 에러를 `_`로 버리지 않는 것이 좋습니다.
- 실패를 먼저 처리하면 정상 흐름이 더 읽기 쉬워집니다.
- 에러 메시지는 문제를 추적할 수 있게 구체적으로 남기는 것이 좋습니다.

`if`문을 배운 뒤 에러 처리를 함께 익히면, Go 코드가 왜 이런 모양으로 자주 작성되는지 훨씬 잘 보입니다.
