## go mod 추가 설명

`go mod`는 Go에서 모듈을 다루기 위한 명령어입니다.  
여기서 모듈은 Go가 코드를 **하나의 프로젝트 단위로 묶고 의존성을 관리하는 방식**이라고 이해하면 됩니다.

`go mod init`을 실행하면 `go.mod` 파일이 생성됩니다.  
이 파일에는 현재 프로젝트의 **모듈 경로**와 **사용하는 Go 버전**이 기록되며, 이후 외부 패키지를 사용하면 필요한 의존성  정보도 함께 관리됩니다.

즉, `go mod`를 사용하는 이유는 단순히 코드를 실행하기 위해서가 아니라,  
현재 코드를 **어떤 프로젝트로 볼 것인지**, 그리고 **어떤 의존성을 어떤 기준으로 사용할 것인지**를 Go에게 분명하게  알려주기 위해서입니다.

작은 예제 파일 하나만 실행할 때는 `go run hello.go`만으로도 충분할 수 있습니다.  
하지만 프로젝트 안에서 패키지를 나누거나, 외부 라이브러리를 추가하거나, 같은 구조를 다시 재현해야 하는  
상황이 생기면 `go.mod` 같은 기준 파일이 필요해집니다.

Go 공식 문서도 `go mod init`이 `go.mod` 파일을 만들고, 그 파일이 코드의 의존성을 추적한다고 설명합니다.  
또한 모듈 경로는 해당 모듈 안에 있는 패키지들의 import 경로 앞부분이 되기 때문에, 모듈은 단순한 설정 파일이 아니라  프로젝트 구조의 기준점 역할도 하게 됩니다. 

### `go mod`란?
---

`go mod`는 모듈을 생성하고 의존성을 정리하고, 수정하는 데 사용되는 Go 명령어 모음입니다.  
이 중 가장 먼저 자주 사용되는 명령어가 `go mod init`입니다.

`go mod init`을 실행하면 `go.mod` 파일이 생성되고, 이 파일이 현재 프로젝트의 모듈 정보를 담기 시작합니다.  
예를 들어 다음 처럼 실행할 수 있습니다.

```bash
go mod init example.com/myproject
```

공식 튜토리얼도 `go mod init`이 `go.mod`파일을 만들고, 처음에는 **모듈 이름과 Go 버전만** 포함한다고 설명합니다.  
이후 의존성이 생기면 그 정보가 추가됩니다.

### 왜 `go mod init`을 해야 할까?
---

`go mod init`을 하는 이유는 단순히 설정 파일 하나를 만들기 위해서가 아닙니다.  
이 명령은 **현재 디렉터리를 하나의 모듈 루트로 정하고**, 그 아래 코드를 같은 프로젝트 범위로 다루겠다고  
선언하는 의미를 가집니다.

공식 블로그도 `go mod init`이 현재 디렉터리를 모듈의 루트로 만들고, 하위 디렉터리  
패키지들은 그 모듈 경로를 기준으로 import 경로를 갖게 된다고 설명합니다.

또 하나의 핵심은 **의존성 추적**입니다.  
외부 패키지를 가져오면 `go.mod`가 어떤 모듈과 버전을 필요한지 기록합니다.  
그래서 같은 프로젝트를 다른 환경에서도 다시 받아도, 가능한 같은 기준으로 빌드할 수 있습니다.

Go 공식 튜토리얼도 이것이 빌드를 재현 가능하게 하고, 사용하는 모듈 버전에 대한 직접적인 제어를 준다고 설명합니다.

### `go.mod` 파일에는 무엇이 들어갈까?
---

Go 공식 문서에 따르면 `go.mod` 파일은 현재 모듈의 속성을 설명합니다.  
대표적으로 들어가는 것은 다음과 같습니다.

모듈 경로, 현재 모듈이 요구하는 최소 Go 버전, 그리고 다른 모듈의 최소 요구 버전입니다.

아주 단순한 시작 상태는 보통 이런 형태입니다.
```go
module example.com/myproject

go 1.24.0
```

여기서 `module`은 현재 프로젝트의 고유한 모듈 경로이고, 이 값은 모듈 내부 패키지들의 import 경로  
앞부분이 됩니다. 공식 문서도 모듈 경로가 패키지 경로의 prefix 역할을 한다고 설명합니다.

### `go mod init`은 어디에서 실행해야 할까?
--- 

`go mod init`은 **파일 하나를 기준으로 실행하는 명령어가 아니라**, 앞으로 하나의 Go 프로젝트로 묶어  
관리할 **폴더**를 기준으로 실행하는 명령어입니다.

공식 블로그도 `go.mod` 파일은 모듈의 루트에만 나타나며, 하위 디렉터리 패키지는 자동으로 그 모듈의 일부가 된다고 설명합니다.  
그래서 하위 폴더마다 다시 `go mod init`을 할 필요도, 보통을 그렇게 하는 것이 바람직하지도 않습니다.

즉, Go 파일이 깊은 하위 폴더에 있어도 그 폴더들이 모두 같은 프로젝트에 속한다면, 가장 위에 있는 공통 폴더에서  
`go mod init`을 실행하면 됩니다. 반대로 특정 하위 폴더만 별도의 프로젝트로 관리하고 싶다면,  
그 폴더에서 별도 모듈을 시작하는 편이 자연스럽습니다.

Go 위키도 보통 하나의 버전 관리 저장소에는 하나의 모듈이 루트에 정의되는 경우가 가장 흔하다고 설명합니다.

### `go mod init` 뒤에는 무엇을 해야 할까?
---

`go mod init`을 실행했다고 해서 프로그램이 바로 실행되는 것은 아닙니다.
이 명령은 **프로젝트를 모듈로 초기화하는 작업**이지, 실행 명령어는 아니기 때문입니다.

그 다음에는 목적에 따라 `go run`, `go build`, `go test` 같은 명령을 사용합니다.  
Go 공식 블로그도 `go mod init` 뒤에는 `go build`, `go test` 같은 패키지 빌드 명령이 필요에 따라 `go.mod`를 갱신한다고 설명합니다.

예를 들어 바로 실행하고 싶다면 

```bash
go run main.go
```

현재 폴더 전체를 실행 대상으로 삼고 싶다면

```bash
go run .
```

실행 파일을 만들고 싶다면
```bash
go build
```

이렇게 구분해서 사용하면 됩니다.

### `go run`과 `go build`의 차이
---

`go run`은 코드를 **바로 실행할 때** 사용하는 명령어입니다.  
학습용 예제나 간단한 동작 확인처럼, 작성한 결과를 빠르게 보고 싶을 때 적합합니다.

반면 `go build`는 코드를 즉시 실행하는 대신 **실행 파일을 생성**합니다.  
겉으로 보기에는 둘 다 결국 프로그램을 실행하는 흐름처럼 보일 수 있지만, 중요한 차이는 **실행 결과물을 남기느냐 그렇지 않느냐**에 있습니다.

예를 들어  
```bash
go run main.go
```
는 코드를 바로 실행하는 방식이고,

```bash
go build -o app
./app
```
는 실행 파일을 먼저 만든 뒤, 그 결과물을 직접 실행하는 방식입니다.

이 때문에 개발이나 학습 중에는 `go run`이 더 자주 사용될 수 있습니다.
코드를 수정한 뒤 바로 결과를 확인하기에 편하기 때문입니다.

하지만 `go build`는 단순히 실행 방식이 하나 더 있는 것이 아니라,
현재 코드 상태를 실행 파일이라는 결과물로 남길 수 있다는 점에서 의미가 있습니다.

즉, 같은 프로그램을 반복해서 실행하거나, 코드가 없는 환경에서 실행하거나, 다른 사람에게 전달할  
실행 파일이 필요할 때는 `go build`가 더 적합합니다.

정리하면, `go run`은 빠르게 확인하기 위한 실행,
`go build`는 실제로 사용할 수 있는 결과물을 만드는 작업에 가깝습니다.

### `go mod tidy`는 언제 사용할까?
---

`go mod tidy`는 현재 코드 기준으로 `go.mod`와 `go.sum`을 정리할 때 사용하는 명령어입니다.  
쉽게 말하면, **지금 코드에 정말 필요한 의존성만 남기고 정리하는 작업**이라고 보면 됩니다.

예를 들어 외부 패키지를 새로 추가했는데 `go.mod`에 아직 반영되지 않았거나,  
반대로 예전에 사용하던 패키지를 코드에서 제거했는데 의존성 정보가 그대로 남아 있을 수 있습니다.  
이럴 때 `go mod tidy`를 실행하면 현재 코드를 기준으로 필요한 의존성은 추가하고, 더 이상 사용하지 않는 의존성은  정리해줍니다.

```bash
go mod tidy
```

즉, go mod tidy는 새로운 기능을 실행하는 명령어라기보다,  
모듈 정보를 현재 코드 상태에 맞게 깔끔하게 맞추는 정리 명령어에 가깝습니다.

보통은 다음과 같은 경우에 자주 사용합니다.

- 외부 패키지를 추가한 뒤 의존성 정보를 정리하고 싶을 때
- 사용하지 않는 패키지를 제거한 뒤 go.mod를 정리하고 싶을 때
-  프로젝트를 커밋하거나 공유하기 전에 의존성 정보를 한 번 정돈하고 싶을 때

따라서 go mod tidy는 매번 반드시 실행해야 하는 명령어라기보다는,  
의존성 정보가 현재 코드와 맞는지 정리하고 싶을 때 사용하는 명령어라고 이해하면 됩니다.

### 정리
---

이번 내용을 통해 `go mod`는 단순히 코드를 실행하기 위한 명령어가 아니라,  
**현재 코드를 하나의 Go 프로젝트로 정의하고, 그 프로젝트의 기준을 정리하기 위한 기능**이라는 점을 확인할 수 있었습니다.

정리하면 다음과 같습니다.

- `go run`  
  코드를 바로 실행할 때 사용합니다.  
  학습용 예제나 간단한 동작 확인에 적합합니다.

- `go build`  
  실행 파일을 생성할 때 사용합니다.  
  결과물을 남기고 싶을 때 적합합니다.

- `go mod tidy`  
  현재 코드 기준으로 `go.mod`와 `go.sum`을 정리할 때 사용합니다.

즉, `go mod`는 지금 당장 실행만 하기 위해 반드시 필요한 기능이라기보다,  
파일이 늘어나고 구조가 생기고 의존성이 추가될수록 **프로젝트를 더 명확하고 안정적으로 관리하기 위해 필요한 기준**이라고 볼 수 있습니다.

처음에는 단순한 설정처럼 보일 수 있지만,  
Go 코드를 파일 단위가 아니라 프로젝트 단위로 바라보기 시작할 때 그 역할이 훨씬 분명하게 느껴집니다.
