## UTF-8과 텍스트 인코딩 추가 설명

문자열을 배우다 보면 `len("가")`가 1이 아니라 3이라는 사실을 만나게 됩니다.  
이 지점에서 중요한 것은 문자열 문법보다 텍스트가 컴퓨터 안에서 어떻게 표현되는지 이해하는 것입니다.

이번 추가 설명은 Go 문자열 자체보다 UTF-8과 인코딩의 기본 감각을 정리합니다.

### 컴퓨터는 문자를 숫자로 저장한다
---

컴퓨터는 문자 자체를 그대로 저장하지 않습니다.  
문자마다 정해진 숫자 값을 부여하고, 그 숫자를 바이트로 저장합니다.

유니코드는 문자에 번호를 부여하는 표준이라고 볼 수 있습니다.  
예를 들어 한글, 영어, 기호, 여러 나라 문자가 각각 고유한 코드 포인트를 가집니다.

Go의 `rune`은 이런 유니코드 코드 포인트를 표현하는 타입입니다.

```go
var r rune = '가'
fmt.Println(r)
```

### UTF-8은 유니코드를 바이트로 저장하는 방식이다
---

유니코드 코드 포인트를 실제 바이트로 저장하는 방식에는 여러 가지가 있습니다.  
그중 Go 문자열에서 가장 흔히 만나는 방식이 UTF-8입니다.

UTF-8은 문자마다 필요한 바이트 수가 다를 수 있습니다.

- 영어 알파벳은 보통 1바이트입니다.
- 한글은 보통 3바이트입니다.
- 일부 이모지는 더 많은 바이트를 사용할 수 있습니다.

그래서 문자열의 바이트 길이와 사람이 보는 글자 수가 다를 수 있습니다.

```go
fmt.Println(len("Go")) // 2
fmt.Println(len("가"))  // 3
```

### Go 소스 파일은 UTF-8을 전제로 한다
---

Go 소스 코드는 UTF-8로 작성되는 것을 전제로 합니다.  
그래서 문자열 리터럴에 한글을 바로 넣을 수 있습니다.

```go
message := "안녕하세요"
```

이 점은 여러 언어를 섞어 다루는 프로그램을 만들 때 편리합니다.  
하지만 문자열을 자를 때 바이트 기준인지 문자 기준인지 항상 구분해야 합니다.

### 바이트 단위 처리가 필요한 경우도 있다
---

네트워크, 파일, 암호화, 압축 같은 영역에서는 문자열보다 바이트 슬라이스를 직접 다루는 경우가 많습니다.

```go
data := []byte("hello")
```

`[]byte`는 텍스트라기보다 원시 데이터에 가깝습니다.  
반대로 사람이 읽는 문자를 다룰 때는 `string`과 `rune` 개념이 더 중요해집니다.

Go에서는 문자열과 바이트 슬라이스를 서로 변환할 수 있습니다.

```go
text := "Go"
bytes := []byte(text)
again := string(bytes)
```

다만 변환은 새 데이터를 만들 수 있으므로, 큰 데이터를 자주 변환할 때는 비용을 생각해야 합니다.

### 깨진 문자는 인코딩 불일치에서 자주 생긴다
---

한글이 깨져 보이는 문제는 보통 데이터를 읽는 쪽과 쓰는 쪽의 인코딩이 다를 때 발생합니다.

예를 들어 어떤 파일이 UTF-8이 아닌 방식으로 저장되어 있는데, 프로그램이 UTF-8이라고 가정하고 읽으면 글자가 깨질 수 있습니다.

Go는 UTF-8과 잘 어울리지만, 모든 외부 데이터가 UTF-8이라는 뜻은 아닙니다.  
파일, API, 데이터베이스에서 가져온 텍스트가 어떤 인코딩인지 확인해야 할 때도 있습니다.

### 정리
---

문자열 장에서는 문법만큼이나 텍스트 인코딩 감각이 중요합니다.

정리하면 다음과 같습니다.

- 컴퓨터는 문자를 숫자와 바이트로 표현합니다.
- 유니코드는 문자에 코드 포인트를 부여하는 표준입니다.
- UTF-8은 유니코드를 바이트로 저장하는 대표적인 방식입니다.
- Go 문자열의 `len()`은 글자 수가 아니라 바이트 수를 반환합니다.
- 바이트 데이터는 `[]byte`, 문자 단위 의미는 `rune`과 연결해서 생각하면 좋습니다.
- 외부 데이터는 인코딩이 다를 수 있으므로 깨진 문자 문제를 의심해야 합니다.

이 내용을 알고 있으면 Go 문자열에서 `byte`, `rune`, `string`이 왜 따로 등장하는지 훨씬 자연스럽게 이해됩니다.
